03 — Architecture technique

Deux bases de données, un choix délibéré

Stack technique
  • Backend — Node.js 20, Express 4, Zod
  • Auth — JWT HS256 + bcrypt coût 12
  • Logs — MongoDB 7 (NoSQL, indexé)
  • Incidents/Auth — SQLite (WAL mode)
  • TLS — Nginx reverse proxy, TLS 1.3
  • Agent — Python 3.11, Windows Event Log
Sécurité by design
  • Helmet.js — en-têtes HTTP durcis
  • CORS restreint — variable d'env
  • Validation Zod — tous les endpoints
  • Fallback SQLite — si MongoDB absent

05 — Infrastructure sécurisée

Durci sur 3 niveaux : réseau, conteneurs, secrets

Réseau Docker
  • Nginx TLS 1.3 en frontal
  • MongoDB non exposé sur l'hôte
  • Réseau interne backend-net
  • MongoDB authentifié (--auth)
  • Redirect HTTP → HTTPS
Conteneurs (CIS Benchmark)
  • Image Alpine (surface minimale)
  • Utilisateur non-root (appuser)
  • npm ci — reproductible
  • mem_limit + cpus par service
  • Frontend read-only
  • Healthcheck Mongo
Secrets & configuration
  • Secrets dans .env (gitignored)
  • Compose via env_file
  • REQUIRE_STRONG_SECRETS au boot
  • JWT ≥ 24 chars, API key ≥ 16 chars
  • Politique de rotation documentée

Chaîne de custody
  • Hash calculé à l'insertion — avant tout stockage
  • Stocké dans MongoDB avec le log
  • Vérifiable à tout moment sans accès DB
  • Violation → audit_log automatique
Fondamental pour la recevabilité des preuves en forensique numérique

14 — Limites & évolutions

Des risques résiduels documentés et assumés

Accepté (prototype)
  • TLS auto-signé vs CA reconnue
  • SQLite non répliqué
  • Pas de rotation automatique des secrets
  • MFA absent sur le compte admin
Prochaines itérations
  • Traefik + Let's Encrypt
  • PostgreSQL + réplication
  • HashiCorp Vault pour les secrets
  • TOTP sur compte admin
  • UEBA (corrélation temporelle)
Ce qui est déjà là
  • ✅ TLS 1.3 fonctionnel
  • ✅ MongoDB authentifié
  • ✅ Pipeline bloquant
  • ✅ Intégrité SHA-256
